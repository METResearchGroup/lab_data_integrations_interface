## Scale Degradation (check latency: table=100,000 vs table=0)

How much slower is the check phase when there are 100K historical URIs vs an empty store?
A coefficient of 1.0 means no slowdown. Each row holds batch size fixed and varies table size only.

S3+Athena uses Prod ms for both numerator and denominator.
S3+DuckDB table=0 short-circuits (no query), so 100K/0 is not comparable to the other
backends. The last column uses 100K/10K instead — both table sizes run real S3 scans.

Batch     S3+SQLite    DynamoDB    S3+Athena (Prod ms)    S3+DuckDB (100K/10K)
100       52.0x        0.87x       0.98x                  1.84x
1,000     18.3x        1.06x       1.02x                  1.32x
5,000     19.9x        0.96x       1.03x                  1.83x
10,000    18.8x        1.02x       1.02x                  1.08x

S3+SQLite degrades 18–52x because it must download the full SQLite file on every run.
As history grows, so does the file, and so does every single pipeline run's latency.

DynamoDB stays at ~1.0x across all table sizes because lookups are O(1) key-value
reads — the table size has no effect on per-item lookup time.

S3+Athena stays at ~1.0x. Cold start overhead (~3,200ms) dominates latency at these
data sizes, masking any scale signal. Degradation would only appear once seen_uris
grows large enough for scan time to exceed cold start time (likely tens of millions
of URIs).

S3+DuckDB shows mild 1.1–1.8x degradation from 10K to 100K (10x more data). Single-
machine S3 download bandwidth is the bottleneck. At hundreds of millions of URIs DuckDB
would hit memory limits; Athena's distributed parallel reads would eventually win.

## Cost Per 1,000 Pipeline Runs (estimated)

S3 + SQLite: $0.0054 per 1,000 runs regardless of batch size or table size.
  - S3 charges a flat fee per GET/PUT request, not per byte — so a larger file
    costs the same per request but takes longer to download (see latency results).
  - Storage: $0.000171/month at 100K URIs (~8MB). Negligible.
  - Data transfer: free if pipeline runs inside the same AWS region as the bucket.
    If running outside AWS (e.g. local machine, HPC), add $0.09/GB per download —
    at 100K URIs that is ~$0.0007/run, growing as history accumulates.

DynamoDB (scales with batch size, assumes all URIs are new):
  batch=100:    $0.14 per 1,000 runs
  batch=1,000:  $1.38 per 1,000 runs
  batch=5,000:  $6.88 per 1,000 runs
  batch=10,000: $13.75 per 1,000 runs

DynamoDB is more expensive per run, but at a daily cadence batch=1,000 costs ~$0.50/year.
DynamoDB cost does not grow with history size — only with batch size.

S3 + Athena: $0.058 per 1,000 runs, flat across all batch sizes and table sizes.
  - Athena scan: $0.048 (10MB minimum billing per query at $5/TB — all 12 combinations
    fall well under 10MB so cost is identical regardless of batch or table size)
  - S3 PUT (batch upload per check + write file): $0.010
  - Storage: $0.00013/month at 100K URIs (~6MB text). Negligible.
  - Note: Athena writes one result file per query to the results S3 prefix.
    Add a lifecycle rule in production to expire these automatically.

S3 + DuckDB: ~$0.010 per 1,000 runs, flat across all batch sizes and table sizes.
  - S3 LIST (1 per check to short-circuit on empty store): $0.005 per 1,000 runs
  - S3 PUT (write file, 1 per run): $0.005 per 1,000 runs
  - S3 GET (httpfs reads seen_uris files): $0.0004 per 1,000 GETs — trivial at low
    file counts; grows slightly as file count increases before compaction.
  - No Athena scan fee. No DynamoDB charges. ~6x cheaper than Athena per 1,000 runs.
  - Storage: same as Athena, ~$0.00013/month at 100K URIs. Negligible.
  - Compute: DuckDB runs on the existing pipeline server — no extra infrastructure cost.
